Preskool.

Frontend.
Login.
1 User opens the web application.
2 Login screen opens.
3 User enters institute number 1111 in the institute number field.
4 User enters username or phone in the username or phone field.
5 User enters password in the password field.
6 User clicks sign in button.
7 Frontend login submit function runs in the login screen.
8 Frontend validates that institute number is not empty and username is not empty and password is not empty.
9 Frontend shows error message on the login screen if any required field is missing.
10 If validation passes frontend calls the api service login function.

API.
Login.
1 Api service builds a request to the auth login endpoint.
2 Request method is post.
3 Request body contains institute number username and password.
4 Frontend sends request without token because user is not authenticated yet.
5 Frontend includes cookies in the request because backend also uses an http only auth cookie.

Backend.
Login.
1 Backend receives the auth login request.
2 Backend validates the request body fields using request validation middleware.
3 Backend queries the master database schools registry to resolve the institute number to a school record.
4 Backend reads school name school type institute number and tenant database name from the schools registry.
5 Backend selects the tenant database connection for Preskool using the resolved tenant database name.
6 Backend queries the users table for an active user where username or email or phone matches the provided identifier.
7 Backend joins user roles to read role name and joins staff to read staff linkage for display name.
8 Backend verifies password using password hash comparison.
9 If user is missing or password is invalid backend returns an authentication error.
10 If password is valid backend creates a signed token that contains user id username role id role name tenant database name and school identity.
11 Backend sets an http only cookie that stores the token.
12 Backend returns a success response with token and user fields including role and school fields.

Database.
Login.
1 Master database table schools is read to resolve institute number.
2 Tenant database table users is read to find matching active user account.
3 Tenant database table user roles is read to resolve role name for the user.
4 Tenant database table staff is read to resolve staff linkage and active status for the user.
5 No records are inserted during login.
6 No records are updated during login.

Final One Line Flow.
User enters institute number username and password and clicks sign in. Frontend validates fields and calls api service login. Api sends post auth login request with credentials. Backend validates input resolves institute to tenant database reads users and user roles verifies password creates token sets auth cookie returns success. Frontend stores token redirects to headmaster dashboard and renders school specific header and menu.

Frontend.
Session and routing.
1 After login frontend stores token in local storage and stores user object in application state.
2 Frontend route guard component checks if authentication has been checked and if session is authenticated.
3 If session is not authenticated frontend redirects to login screen.
4 If session is authenticated frontend renders the main layout feature container and the requested route screen.
5 Frontend redirects to headmaster dashboard route for admin role.

API.
Session and routing.
1 For all subsequent requests api service reads token from local storage.
2 Api service sends the token in the Authorization header using Bearer token format.
3 Api service also includes credentials to send and receive the auth cookie.

Backend.
Session and routing.
1 Backend protects all api routes except auth login and health endpoints.
2 Backend reads token from http only cookie or from Authorization header.
3 Backend verifies token signature and expiration.
4 Backend attaches decoded user payload to the request context.
5 Backend binds the request to the tenant database connection using the tenant database name inside the token.
6 Backend enforces active account rules for student and staff accounts using an active account middleware.

Database.
Session and routing.
1 No database changes occur for session verification.
2 All subsequent queries run against the tenant database selected from the token.

Final One Line Flow.
Frontend navigates to protected screen. Api sends request with Bearer token and cookie. Backend verifies token binds tenant database checks active account continues to controller. Controller reads or writes tenant tables returns response. Frontend renders updated screen state.

API.

1 Endpoint is students.

Backend.
Create student.
1 Backend verifies token and checks admin role for create student.
2 Backend validates request body required fields.
3 Backend starts a database transaction.
4 Backend checks for duplicate admission number in students table.
5 Backend inserts student record into students table using schema tolerant logic for religion field and address fields.
6 Backend creates a linked user account for the student when admission number or phone or email is provided.
7 Backend updates students user id foreign key to link student to user.
8 If parent fields are provided backend inserts a parents record linked by student id.
9 Backend updates students parent id foreign key to link student to parent.
10 Backend creates a linked user account for parent when parent email or phone is provided.
11 Backend updates parents user id foreign key to link parent to user.
12 If guardian fields are provided backend inserts guardians record linked by student id.
13 Backend updates students guardian id foreign key to link student to guardian.
14 Backend creates a linked user account for guardian when guardian phone or email is provided.
15 Backend updates guardians user id foreign key to link guardian to user.
16 Backend synchronizes address into addresses table for the student user id when address values exist.
17 Backend commits the transaction and returns created student record.

Database.
Create student.
1 Students table insert occurs for the student master record.
2 Users table insert may occur for student login account.
3 Students table update occurs to set students user id foreign key.
4 Parents table insert may occur for parent record linked to student id.
5 Students table update may occur to set students parent id foreign key.
6 Users table insert may occur for parent login account.
7 Parents table update may occur to set parents user id foreign key.
8 Guardians table insert may occur for guardian record linked to student id.
9 Students table update may occur to set students guardian id foreign key.
10 Users table insert may occur for guardian login account.
11 Guardians table update may occur to set guardians user id foreign key.
12 Addresses table insert or update may occur for student user id with role student and person id link to student id.

Final One Line Flow.
User fills add student form and clicks save. Frontend validates and calls create student. Api sends post students with token. Backend authorizes admin validates starts transaction checks admission number inserts student creates student user links student inserts parent and guardian links and creates users syncs addresses commits returns success. Frontend shows success and refreshes list.
